# 📄 Contract RAG — Document QA with Hybrid Retrieval

A RAG (Retrieval-Augmented Generation) prototype for contract and long-document QA, featuring multi-format parsing, normalized text pipeline, structure-aware chunking, and hybrid retrieval with a built-in debug panel.

This is not a toy demo — it's designed to demonstrate practical retrieval quality optimization in a realistic document QA scenario.

## ✨ Core Features

- **Multi-format parsing** — PDF, TXT, DOCX with unified output structure
- **Normalized text pipeline** — PDF text cleaning (broken-line merging, whitespace normalization, header/footer deduplication) before chunking
- **Structure-aware chunking** — document router selects `page_fixed` or `paragraph` strategy based on detected document category
- **Hybrid retrieval** — dense (embedding) + sparse (BM25) search, fused via Reciprocal Rank Fusion (RRF)
- **Retrieval debug panel** — UI shows dense score, BM25 score, RRF fused rank, and whether each chunk entered the final answer context
- **Anti-hallucination constraints** — LLM answers strictly from retrieved context; refuses to answer when context is insufficient

## 📑 Document Processing Strategy

### Parsing

All formats output a unified page structure:

```python
{
    "page": 1,
    "text": "...",               # raw extracted text
    "normalized_text": "...",    # cleaned text optimized for retrieval
    "file_type": "pdf"           # pdf / txt / docx
}
```

- **PDF** — extracted per-page via `pdfplumber`. Raw text preserved in `text`; `normalized_text` is generated by `text_normalizer.py` with:
  - Broken-line merging (common in PDF extraction)
  - Whitespace normalization
  - Excess blank line collapsing
  - Lightweight header/footer deduplication (cross-page repetition heuristic)
- **TXT / DOCX** — light cleaning applied to both `text` and `normalized_text`

### Chunking

Chunking operates on `normalized_text` (falls back to `text` if absent). Strategy is selected by `document_router.py`:

| Document Category | Chunk Strategy | Description |
|---|---|---|
| `structured_pdf` | `page_fixed` | Page-level context preserved; split within page if too long |
| `natural_pdf` | `paragraph` | Paragraph-first splitting with merge + overlap |
| `natural_text` | `paragraph` | Same paragraph strategy for TXT / DOCX |

Metadata (`page`, `source`, `file_type`, `chunk_index`, `doc_category`, `chunk_strategy`) is attached to every chunk and stored in ChromaDB.

## 🔍 Retrieval Strategy

### Hybrid Retrieval (default)

Every query goes through both retrieval paths and fuses the results:

1. **Dense retrieval** — embedding similarity via ChromaDB (`ZhipuAI embedding-3`)
2. **BM25 sparse retrieval** — keyword matching via in-memory `rank-bm25` index
3. **RRF fusion** — Reciprocal Rank Fusion (`k=60`) merges the two ranked lists into a single result

The BM25 index is automatically rebuilt after each document index/delete operation. On app startup, it syncs from the existing vector store.

### Retrieval Debug Panel

After each query, the UI displays an expandable debug panel for every retrieved chunk:

| Field | Description |
|---|---|
| `source` | Document filename |
| `page` | Page number |
| `file_type` | pdf / txt / docx |
| `dense_rank` | Rank from dense retrieval |
| `dense_score` | Distance score from dense retrieval |
| `bm25_rank` | Rank from BM25 retrieval |
| `bm25_score` | BM25 relevance score |
| `rrf_score` | Fused RRF score |
| `fused_rank` | Final rank after fusion |
| `in_context` | Whether this chunk was included in the LLM prompt |

## ⚠️ Current Limitations & Trade-offs

- **BM25 is in-memory only** — rebuilt from vector store on startup and after each index change. Suitable for small-to-medium chunk sets (thousands), not for large-scale production.
- **PDF text cleaning is heuristic** — header/footer detection uses cross-page repetition counting; not a full layout parser or OCR pipeline.
- **No OCR support** — scanned PDFs or image-based PDFs are not handled.
- **Streamlit-based** — single-process demo architecture, not a production service.
- **No persistent BM25 index** — acceptable for prototype scope; a production system would use a proper search engine.
- **RRF k=60** — standard default; not tuned per dataset.
